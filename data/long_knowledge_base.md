# 本地 RAG 长知识库测试资料

## 1. RAG 的基本思想

RAG 是 Retrieval-Augmented Generation 的缩写，通常翻译为检索增强生成。它的核心思想不是让大模型记住所有资料，而是在回答前先从外部知识库中检索相关内容，再让模型基于这些内容组织答案。

在本项目中，外部知识库就是 data 目录里的本地文件。文件可以是 Markdown、普通文本，也可以是文字型 PDF。资料进入知识库前，需要被读取、切分、向量化，并写入 storage 目录中的索引文件。

RAG 的优势在于可以使用本地私有资料，不必把所有知识都写进模型参数。它也方便更新：当资料变化时，只需要重新运行 ingest.py 建立索引，不需要重新训练模型。

## 2. 本项目的本地模型配置

本项目使用 Ollama 在本机运行模型，不消耗 OpenAI API 额度。聊天模型使用 qwen2.5:1.5b，主要负责根据检索到的上下文生成中文回答。embedding 模型使用 nomic-embed-text，主要负责把文本片段和用户问题转换成向量。

这两个模型分工不同。qwen2.5:1.5b 并不负责建立向量索引，nomic-embed-text 也不负责生成自然语言回答。前者更像回答者，后者更像资料检索时的语义编码器。

如果 Ollama 服务没有启动，项目在建索引或提问时会连接不到 http://127.0.0.1:11434。遇到这种情况，应该先打开 Ollama 应用，或者在终端运行 ollama serve。

## 3. 文档读取与 PDF 支持

普通文本文件可以直接被 SimpleDirectoryReader 读取。PDF 文件比较特殊，因为 PDF 是页面排版格式，不一定天然包含可复制文本。本项目使用 PyMuPDF 逐页提取 PDF 文本，并把页码写入 metadata。

如果 PDF 是文字型 PDF，PyMuPDF 通常可以提取出正文。参考资料输出中会显示文件名、路径、页码、相似度和片段预览。这样可以检查回答是否真的来自本地资料。

如果 PDF 是扫描版图片，页面中可能没有可提取文本。当前项目会打印警告，提示某个 PDF 的某一页没有提取到文字，并跳过该页。扫描版 PDF 需要 OCR，但 OCR 不属于当前阶段的重点。

## 4. 切片策略

文档进入向量索引前会被切成较小的片段，这个过程通常叫 chunking，中文可以叫切片或分块。切片的原因是长文档太大，直接整体向量化和检索效果通常不好。

chunk_size 表示每个片段的大致长度。chunk_overlap 表示相邻片段之间保留多少重叠内容。重叠的作用是避免一句话或一个概念刚好被切断后丢失上下文。

如果 chunk_size 太小，检索会更精细，但片段可能缺少上下文。如果 chunk_size 太大，每个片段包含的信息太杂，检索命中后模型可能拿到很多无关内容。当前项目把 chunk_size(切片大小)设置为 320，把 chunk_overlap（切片的重叠部分）设置为 60，用于教学和观察。

## 5. 检索数量 top_k

similarity_top_k 表示每次提问时从索引里取回几个最相似的片段。top_k 为 1 时，系统只拿最相关的一个片段，输出更集中，但如果第一名片段不完整，就容易漏掉答案。top_k 为 3 时，系统会拿三个片段，召回率更高，但也更可能带入无关内容。

检索数量不是越大越好。资料少时 top_k 可以小一些；资料多且问题复杂时，可以适当增大。真实项目中通常需要用测试问题反复比较不同 top_k 的效果。

## 6. 相似度分数

相似度分数来自问题向量和资料片段向量之间的匹配程度。分数越高，通常说明这个片段和问题越相关。但相似度不是正确率，也不是置信度。

如果最高相似度很低，说明知识库里可能没有足够相关的资料。这种情况下让模型继续回答，容易产生幻觉。后续可以加入相似度阈值，例如最高分低于某个数值时直接拒答。

## 7. 本地知识库的工程流程

使用本地知识库时，一般流程是：把资料放入 data，运行 python ingest.py，等待索引写入 storage，再运行 python ask.py 提问。修改资料后必须重新运行 ingest.py，否则系统仍然会使用旧索引。

storage 目录是索引结果，不是原始资料。data 目录才是原始知识库。如果 data 和 storage 不一致，以 storage 为准进行问答，所以更新资料后忘记重建索引是常见错误。

## 8. 容易混淆的概念

RAG 不是微调。微调是改变模型参数，让模型本身学到某些风格或任务模式。RAG 是在回答时检索外部资料，模型参数不变。

embedding 不是聊天模型。embedding 模型输出的是向量，用来衡量语义相似度。聊天模型输出的是自然语言，用来生成答案。

索引不是数据库全文搜索的简单替代。向量检索擅长语义相似，但不一定擅长精确字符串匹配。如果需要精确查找编号、公式或表格字段，可能要结合关键词检索。

## 9. 一个刻意增加难度的测试段落

这里加入一些相近但不同的概念，用于测试检索是否足够准确。确定性时延关注端到端时延上界和时延稳定性，低时延通信关注平均时延或短时延体验，可靠通信关注传输成功率和错误概率。本地知识库问答关注资料检索和答案可追溯性。

如果问题问“本项目的 chunk_overlap 是多少”，答案应该来自切片策略部分，而不是来自低时延通信部分。如果问题问“修改资料后为什么要重新 ingest”，答案应该来自工程流程部分，而不是来自模型配置部分。

## 10. 当前阶段建议

当前阶段最重要的是观察切片和检索。你应该尝试修改 chunk_size、chunk_overlap 和 similarity_top_k，然后比较同一个问题返回的参考资料是否变化。

不要只看最终回答，还要看参考资料的文件名、页码、相似度和片段内容。RAG 的学习重点在于理解系统检索了什么，而不是只看模型说了什么。
